① 一把尺子：这段代码该放哪？ ↑ 顶部

你会乱，是因为你在问「我该建几个目录」——这个问题没法回答。换成问这一句，答案自己就冒出来了：

「这段代码是关于『什么』的？」

一个包（目录）= 一个职责 = 一个能自圆其说的名词 / 动词

把 stage07 要写的东西挨个这么问一遍，它自己就分好了：

你要写的东西它是关于「什么」的放进哪个包
Result 结构体数据长什么样（名词）model/
probe()、并发采集探测这个能力（动词）prober/
解析 / 生成 Prometheus 文本格式转换这个能力（动词）promfmt/
/metrics 端点把 HTTP 世界翻译成业务调用handler/
组装 + 启动接线员，只搭线不干活main.go

💡 这跟你 stage06 刚过的 model → store → handler → main 是同一个骨架，只是把 store（存数据）换成了 prober（采数据）。 你不是在学新结构，你是在复用同一个结构。

② 依赖只能有一个方向（不乱的关键） ↑ 顶部

铁律：底层不认识上层，箭头绝不回头。 model 谁都能依赖，但它不依赖任何人（所以它最纯，只有结构体）； main 认识所有人（它是接线员）。一旦 model 反过来 import handler，Go 直接编译报错 import cycle。

main接线员 · 组装启动
↓ import
handlerHTTP → 业务调用
↓ import
prober探测能力
promfmt格式转换能力
↓ import
model数据长什么样 · 谁都能用

依赖像水，只能从高处（main）往低处（model）流。model 是地基，地基不会去依赖楼顶。

③ 什么时候该拆？别过度设计 ↑ 顶部

新手容易两个极端：要么全塞一个 main.go，要么一上来拆十个目录。正确姿势是：起步就一个 main.go，出现下面信号之一才拆。

🔸 信号一：一个文件里塞了两坨互不相干的东西（探测逻辑 + 格式解析）。

🔸 信号二：你想单独测某段（只测解析函数，不想连 HTTP 一起启动）。

🔸 信号三：有段代码想复用（probe 被「命令行采集」和「/metrics」都要用）。

🔧 类比运维：一个 100 行的部署脚本无所谓；但当它同时管「装包 + 配 nginx + 起服务」三件事， 你自然会拆成 Ansible 的三个 role。拆的依据是职责，不是行数。

④ 五个文件，逐个拆解 ↑ 顶部

stage07-metrics-collector/
├── go.mod # module metrics-collector
├── main.go # 接线员：造零件 → 接线 → 启动
├── model/model.go # Result 结构体（血液）
├── prober/prober.go # Probe 单探测 + ProbeAll 并发采集
├── promfmt/promfmt.go # Parse 解析 + WriteMetrics 生成
└── handler/handler.go # /metrics 端点

model —— 流经所有层的「血液」

它只有结构体、没有逻辑，所以站在依赖链最底层：prober 产出它、handler 端着它、promfmt 消费它。有个共享的数据类型，各层之间才有的可传。

type Result struct {
    Target     string        // 目标 URL
    Success    bool          // 连得通 且 状态码 2xx
    StatusCode int           // HTTP 状态码（失败时为 0）
    Duration   time.Duration // 响应耗时
    Err        string        // 错误信息（成功时为空）
}

prober —— 探测这个「能力」

把 *http.Client 攥在结构体字段里，全程序共用一个（复用 TCP 连接池）。这一关三条铁律全在 Probe 里：

func (p *Prober) Probe(url string) model.Result {
    start := time.Now()

    resp, err := p.client.Get(url)         // ① 超时靠 client，不用裸 http.Get
    if err != nil {
        // 网络层就挂了：把失败也记成一条结果 → 错误隔离，不拖累别人
        return model.Result{Target:url, Success:false,
            Duration:time.Since(start), Err:err.Error()}
    }
    defer resp.Body.Close()               // ② 确认 err==nil 后再 defer Close
    _, _ = io.Copy(io.Discard, resp.Body) // 读干净，连接才能复用

    return model.Result{
        Target:     url,
        Success:    resp.StatusCode >= 200 && resp.StatusCode < 300, // ③ err==nil ≠ 成功
        StatusCode: resp.StatusCode,
        Duration:   time.Since(start),
    }
}

promfmt —— 格式转换这个「能力」

Parse 是 stage04 的活（bufio.Scanner 逐行、跳过 #、strconv 转数字）。WriteMetrics 有个小心机：参数用 io.Writer 而不是 http.ResponseWriter——

🧠 要谁的最小能力，就写谁。 这里只需要「能写」，所以写 io.Writer。 好处：它既能写进 HTTP 响应，也能写进 bytes.Buffer 做单元测试。（呼应 stage02 的接口思想）

handler —— 把 HTTP 翻译成业务调用

handler 不自己 http.Get（那是 prober 的活），也不自己拼字符串（那是 promfmt 的活）。它只做「翻译」：

func (h *Handler) Metrics(w http.ResponseWriter, r *http.Request) {
    results := h.prober.ProbeAll(h.targets)          // 干活外包给 prober
    w.Header().Set("Content-Type", "text/plain; charset=utf-8")
    promfmt.WriteMetrics(w, results)              // 格式化外包给 promfmt
}

注意 handler 依赖的是接口 ProbeAller（"能给我一批结果就行"），不是具体的 *prober.Prober。这样想换实现 / 测试时，handler 一行都不用改。

main —— 接线员，一行业务都不写

func main() {
    p := prober.New(3 * time.Second)      // 造零件
    h := handler.New(p, targets)          // 接线：把探测器和目标清单注入 handler
    mux := http.NewServeMux()
    mux.HandleFunc("/metrics", h.Metrics)  // 接线：路由
    http.ListenAndServe(":8081", mux)     // 启动
}

以后想加个目标、换个超时、改个端口——只改 main，不动任何一层。这就是「接线员」模式的价值。

⑤ 一次请求的旅程（动画） ↑ 顶部

这就是治你「串不起来」的正主。点开始，看一次 curl :8081/metrics 是怎么在各层之间流动的：

1
CURL
curl :8081/metrics —— 有人来抓指标了。请求打到 8081 端口。
2
MAIN（早就接好线了）
启动时 mux.HandleFunc("/metrics", h.Metrics) 已把路由和 handler 绑好， 请求被路由到 h.Metrics。
3
HANDLER
「有人要指标了」→ 调 prober.ProbeAll(targets)，把「探测」这件事外包出去。
4
PROBER
并发开 3 个 goroutine，各自 Probe(url)，各拿回一条 model.Result。Mutex 护着塞进 []Result。
5
PROBER → HANDLER
wg.Wait() 等三个都跑完，把 []model.Result 交回 handler。
6
HANDLER → PROMFMT
handler 把结果交给 promfmt.WriteMetrics(w, results)，把「格式化」也外包出去。
7
PROMFMT
逐条格式化成 probe_success{target="..."} 1，直接写进 w（就是 HTTP 响应流）。
8
CURL
收到合法的 Prometheus 文本。旅程结束。 ✓ 完成
血液 = model.Result 流经所有层 main 只接线 每层只干一件事，把别的外包

看懂这张图，你就抓住了「串联」的本质：

① model.Result 是流经所有层的血液 —— 有个共享数据类型，各层才有的可传，所以 model 要最底层。

② 每层只干自己那件事，把别的外包 —— handler 不联网、不拼字符串，它只做翻译。每层都薄到一眼看懂，合起来就不乱。

③ main 负责接线，别的层负责干活 —— 谁 new 谁、谁传给谁，全在 main 一处发生，业务逻辑一行都不在 main 里。

⑥ 这一关的四个新坑 ↑ 顶部

坑 1 · 别用裸 http.Get()

它没有超时。目标卡住，你的 goroutine 就永远挂在那，越堆越多，最后拖垮整个采集器。

生产铁律：任何网络请求都要有超时。 用 &http.Client{Timeout: 3*time.Second}。

坑 2 · err == nil ≠ 成功

HTTP 500 / 404 时，err 依然是 nil！因为「网络通了、拿到响应了」这件事本身没出错。

if err == nil && resp.StatusCode >= 200 && resp.StatusCode < 300 {
    success = true   // 必须另外看状态码
}

类比：电话打通了（err==nil）≠ 对方答应了你（2xx）。

坑 3 · resp.Body 必须 Close，且顺序有讲究

resp, err := client.Get(url)
if err != nil {
    return              // ← err!=nil 时 resp 可能是 nil，不能 defer Close
}
defer resp.Body.Close()   // ← 确认 err==nil 之后再 defer

不 Close = 连接泄漏（像文件不 Close）。顺序反了 = 对 nil 调 Close 直接 panic。

坑 4 · goroutine 闭包变量陷阱 ⭐

// ❌ 所有 goroutine 共享同一个 url，循环跑完 url 已是最后一个
for _, url := range targets {
    go func() { probe(client, url) }()   // 大概率都抓最后一个！
}
// ✅ 把 url 当参数传进去，每个 goroutine 拿到自己的副本
for _, url := range targets {
    go func(u string) { probe(client, u) }(url)
}

注：Go 1.22+ 改了循环变量语义，新版可能不踩。但显式传参永远安全，别赌版本。
