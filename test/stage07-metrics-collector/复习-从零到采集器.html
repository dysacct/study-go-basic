0. 先别急，为什么你会「学了就忘」

这不是智商问题，是「输入」和「输出」的次数不对等。你读一遍代码是输入，但真正刻进脑子的是输出——自己动手敲、跑、报错、改对。你运维里 iptables 记得牢，不是因为你聪明，是因为你敲了几百遍。代码一模一样。

积木 1 · 结构体 struct来自 stage01

一句话：struct 就是「把几个相关的字段打包成一个盒子」，给这个盒子起个类型名，以后就能整盒整盒地传。

最小可跑代码

// brick1.go
package main

import "fmt"

// 定义一个「盒子」类型：一台服务器的信息
type Server struct {
    Name   string
    IP     string
    Status string
}

func main() {
    // 造一个盒子实例
    s := Server{Name: "web-01", IP: "10.0.0.5", Status: "running"}

    fmt.Println(s.Name)      // 用 . 取字段 → web-01
    fmt.Println(s.Status)    // running
    fmt.Printf("%+v\n", s)   // 整盒打印 → {Name:web-01 IP:10.0.0.5 Status:running}
}
运维类比：struct 就像一台机器的「资产登记表」——主机名、IP、状态填在固定的格子里。你不会把这三样散着记，你会记成「一台机器 = 这么一张表」。struct 就是那张表的模板。
基础题 1：定义一个 Result 结构体，含三个字段：Target string、Success bool、Cost int。在 main 里造一个实例（Target 填 "http://localhost:8080"，Success 填 true，Cost 填 23），然后用 fmt.Printf("%+v\n", ...) 整盒打印出来。

积木 2 · 切片 slice + append来自 stage01

一句话：slice 是「一串同类型的盒子」，可变长。append 往末尾塞一个，range 挨个遍历。

最小可跑代码

// brick2.go
package main

import "fmt"

type Server struct {
    Name   string
    Status string
}

func main() {
    // 空切片：一串 Server，目前 0 个
    servers := []Server{}

    // append 往末尾塞。注意：必须接回原变量 servers = append(...)
    servers = append(servers, Server{Name: "web-01", Status: "running"})
    servers = append(servers, Server{Name: "db-01", Status: "stopped"})

    fmt.Println("共", len(servers), "台")   // 共 2 台

    // range 遍历：i 是下标，s 是每个元素的副本
    for i, s := range servers {
        fmt.Println(i, s.Name, s.Status)
    }
}
最容易忘的一点：append 的返回值必须接回去——servers = append(servers, x)。只写 append(servers, x) 不接，等于白干。原因：切片满了会「搬家」到新地址，不接回你手里还是旧的。
运维类比：slice 就像 hosts 文件里的一串主机，一行一台，能随时往下加。range 就是 for h in $(cat hosts) 挨个过。
基础题 2：用切片装 3 个 Server（自己起名字和状态），然后 range 遍历，只打印状态是 "running" 的机器名。（练 append + range + if 判断）

积木 3 · goroutine + WaitGroup来自 stage03

一句话：go 函数() 就是「让这个函数自己去后台跑，我不等它」；但 main 一结束程序就退了，所以要用 WaitGroup 数着「还有几个没跑完」，Wait() 卡住等全部完成。

最小可跑代码

// brick3.go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup   // 计数器，初始 0

    for i := 1; i <= 3; i++ {
        wg.Add(1)           // 每开一个 goroutine，计数 +1

        go func(n int) {    // go = 后台跑；n 当参数传进来（关键，见下）
            defer wg.Done() // 这个 goroutine 干完，计数 -1
            fmt.Println("worker", n, "干活")
        }(i)                // 立刻把当前的 i 传进去
    }

    wg.Wait()               // 卡在这，直到计数归 0（3 个都 Done 了）
    fmt.Println("全部干完")
}
经典坑（一定要记）：goroutine 里要用循环变量 i，一定当参数传进去 go func(n int){...}(i)，别在闭包里直接用 i。否则 3 个 goroutine 可能都拿到最后一个值。运维类比：你 for h in hosts; do ssh $h ... & done 时，也得确保每个后台任务拿到的是「它自己那台」主机名。
运维类比：go func() 就是 shell 里命令后面加 & 丢后台。wg.Wait() 就是 wait 命令——等所有后台任务都结束再往下走。
基础题 3：并发启动 5 个 goroutine，每个打印 "ping N"（N 是 1 到 5）。用 WaitGroup 等 5 个全部打印完，最后打印一句 "全部 ping 完"。（记得把 N 当参数传进去）

积木 4 · Mutex 并发安全来自 stage06

一句话：多个 goroutine 同时改同一份数据（比如往同一个切片 append）会打架、丢数据。Mutex（互斥锁）让它们排队，一次只放一个进去改。

最小可跑代码

// brick4.go —— 用 go run -race brick4.go 跑，验证无竞争
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup
    var mu sync.Mutex          // 一把锁
    nums := []int{}            // 共享切片，大家一起往里塞

    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()

            mu.Lock()          // 进门，锁上。别人只能在门外等
            nums = append(nums, n)
            mu.Unlock()        // 出门，开锁。下一个才能进
        }(i)
    }

    wg.Wait()
    fmt.Println("塞了", len(nums), "个")   // 稳定是 100；不加锁可能少于 100
}
两条铁律：① Lock() 和 Unlock() 必须成对，常写成 mu.Lock(); defer mu.Unlock() 防漏。② 锁里只放「改数据」这一下，别把联网、打印这些慢活锁进去——锁占越久，别人等越久。
运维类比：Mutex 就是 flock 文件锁。多个脚本同时写一个文件会写乱，加 flock 让它们排队。go run -race 就是「照妖镜」，专抓没加锁的并发读写。
基础题 4：把基础题 3 升级——开 100 个 goroutine，每个往一个共享的 []int 里 append 自己的编号，用 Mutex 保护。最后打印 len，应该稳定是 100。用 go run -race brick4.go 跑，确认没有 DATA RACE 警告。

积木 5 · HTTP 客户端stage07 新知识

一句话：前面写的都是「服务端」（等别人来）。客户端反过来——你主动去 GET 别人的接口。核心就三行：建带超时的 client、client.Get(url)、defer resp.Body.Close()。

最小可跑代码

// brick5.go —— 先启动 stage06 的服务（:8080），再跑这个
package main

import (
    "fmt"
    "io"
    "net/http"
    "time"
)

func main() {
    // ① 建一个带超时的 client（别用裸 http.Get，它没超时会永远卡）
    client := &http.Client{Timeout: 3 * time.Second}

    // ② 主动发 GET
    resp, err := client.Get("http://localhost:8080/health")
    if err != nil {
        // 连不上、超时、DNS 失败都走这（网络层错误）
        fmt.Println("请求失败:", err)
        return
    }
    defer resp.Body.Close()   // ③ 必须关，否则连接泄漏

    // ④ 读响应体
    body, _ := io.ReadAll(resp.Body)

    fmt.Println("状态码:", resp.StatusCode)   // 200
    fmt.Println("内容:", string(body))        // OK
}
最反直觉的一点：err == nil 不代表成功！HTTP 500 / 404 时 err 照样是 nil（因为「网络通了、拿到响应了」本身没出错）。要不要算成功，得另外看 resp.StatusCode 是不是 2xx。类比：电话打通了（err==nil）≠ 对方答应了你（2xx）。
运维类比：这就是 Go 版的 curl -m 3 http://...。-m 3 就是那个 3 秒超时；查 $? 和 HTTP 状态码，就是分别看 err 和 resp.StatusCode。
基础题 5：先启动 stage06 的服务当靶子（cd stage06-concurrent-safe && go run .）。然后写程序：用带超时的 client 探测 http://localhost:8080/health，打印状态码；再探测一个故意连不上的地址 http://localhost:9999/nope，确认它走 err != nil 分支打印错误、程序不崩。（练超时 client + err 分支 + defer Close）

积木 6 · 文本解析来自 stage04

一句话：把一段多行文本，用 bufio.Scanner 逐行读，strings 切一切，strconv 把字符串数字转成真数字。这是日志分析、指标解析的通用手艺。

最小可跑代码

// brick6.go
package main

import (
    "bufio"
    "fmt"
    "strconv"
    "strings"
)

func main() {
    text := `# 这是注释行，要跳过
cpu 80
mem 60

disk 45`

    // 把字符串包成可逐行扫描的 Scanner
    scanner := bufio.NewScanner(strings.NewReader(text))

    total := 0
    for scanner.Scan() {
        line := strings.TrimSpace(scanner.Text())   // 去掉首尾空白

        // 跳过空行 和 # 注释行
        if line == "" || strings.HasPrefix(line, "#") {
            continue
        }

        // 按空格切成若干段：["cpu", "80"]
        parts := strings.Fields(line)
        if len(parts) < 2 {
            continue
        }

        name := parts[0]
        value, err := strconv.Atoi(parts[1])   // 字符串 "80" → 数字 80
        if err != nil {
            continue
        }

        fmt.Printf("%s = %d\n", name, value)
        total += value
    }
    fmt.Println("总和:", total)   // 80+60+45 = 185
}
运维类比：这就是 Go 版的 grep -v '^#' file | awk '{print $1, $2}'。bufio.Scanner = 逐行读，strings.Fields = awk 按空格切列，strconv.Atoi = 把文本数字变成能算的数。
基础题 6：给定下面这段文本（硬编码成字符串），逐行解析，跳过 # 注释和空行，打印每个指标名和值，并算出所有值的总和。用 strconv.ParseFloat(s, 64) 转成 float64（因为有小数）。

node_load 1.5
node_temp 42.3

合体 · 把 6 块积木垒成一个采集器（大题）

到这一步，如果前面 6 块你都亲手跑通了，这道大题没有一个新知识点——它就是把你已经会的东西拼起来。别怕，跟着骨架填空。

要做什么

写一个 mini 采集器：给一批 URL，并发去探测每一个，把「通不通、状态码、耗时」记成结果，最后打印汇总。这正好是 stage07 的任务 1 + 任务 2 的核心。

这道大题用到哪几块积木： 积木1 struct → 定义 Result（装一次探测的结果） 积木5 HTTP客户端 → probe() 探测单个目标 积木3 goroutine → 并发探测多个目标 积木4 Mutex → 安全地把结果 append 进共享切片 积木2 slice → 用切片收集所有 Result，range 打印 （积木6 文本解析这道大题用不上，它是 stage07 后半段任务 3 的）
